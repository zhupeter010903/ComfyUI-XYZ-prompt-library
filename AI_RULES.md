# AI_RULES — XYZ Image Gallery 编程行为约束

> 本文件用于**严格约束** AI 助手（包括但不限于 Cursor / Claude / Copilot）在本仓库中的编码行为。
> 适用范围：`ComfyUI-XYZNodes/` 下所有与 Gallery 子模块相关的开发活动。
>
> **优先级（自上而下，强制顺序）**：
> 1. 本文件 (`AI_RULES.md`) 的硬性约束
> 2. `PROJECT_SPEC.md`（功能 / 非功能规格 = 事实来源）
> 3. `ARCHITECTURE.md`（模块边界与数据流）
> 4. `TASKS.md`（任务拆解 / 验收标准 / 依赖关系）
> 5. 用户在当前对话中的明确指令
>
> **当上述文档之间出现冲突时**：必须**停下来提问**，禁止自行裁决。
>
> **关键词约定**（参照 RFC 2119）：
> - **MUST / 必须 / 严禁** — 硬性约束，违反即为错误。
> - **SHOULD / 应当** — 强烈建议，违反需在回复中显式说明理由。
> - **MAY / 可以** — 可选行为。

## 0. 总则（Meta-Rules）

- **R0.1** — AI **必须**在动手前确认：当前正在执行的是 `TASKS.md` 中**哪一个任务 ID**（例如 `T07`）。如果用户没有指定，**必须先问**，**严禁**自行假设。
- **R0.2** — AI **必须**在动手前列出本次将**读 / 写 / 创建 / 删除**的文件清单，等待用户确认或自行明确边界后再执行。
- **R0.3** — AI **严禁**在一次回复中"顺便"完成多个任务、补全 TODO、清理无关代码、修复无关 bug 或"优化"无关模块。
- **R0.4** — 当任何下列触发条件出现时，AI **必须停止执行并提问**：
  - 任务边界不清晰；
  - 改动会影响未列出的文件；
  - 实现路径与 `PROJECT_SPEC.md` / `ARCHITECTURE.md` 存在出入；
  - 用户描述的需求与文档矛盾；
  - 现有代码与文档不一致，无法判断以哪一方为准。
- **R0.5** — **严禁**编造规格、API、字段、表结构、文件路径、模块名。任何不在文档中、不在代码中、不在用户指令中的设定，都**必须**先问。

## 1. 单任务原则（One Task At A Time）

> 对应需求 1：**只能实现当前任务，不允许跨任务实现功能**。

- **R1.1** — 每次会话**只允许**实现 `TASKS.md` 中的**单个任务 ID**（例：仅 T07，而非 T07 + T08）。
- **R1.2** — **严禁**实现"下一个任务可能会用到的字段 / 方法 / 路由 / 表列"。即使这些内容稍后必然要写，也必须留到对应任务再做。
- **R1.3** — **严禁**为"将来扩展"预留接口、参数、抽象层、配置项、事件类型。所有"以后再说"的代码都视作**未授权代码**。
- **R1.4** — 如果当前任务的实现**严格依赖**尚未完成的前置任务，**必须停下提问**，由用户决定是先做前置还是改派任务，**严禁**自行实现前置任务。
- **R1.5** — 任务完成的判定标准 = `TASKS.md` 中该任务"测试 / 验收"小节列出的**全部**条目通过；不可多做、不可少做。
- **R1.6** — 完成任务后**必须**给出"已实现 / 未实现 / 故意未做"的明确清单，便于用户对账。

## 2. 文件改动白名单（File Scope Lock）

> 对应需求 2：**不允许修改未被允许的文件**。

- **R2.1** — AI **必须**先声明本次任务的**文件白名单**（新建文件 + 修改文件），未列入白名单的文件**严禁**改动。
- **R2.2** — 文件白名单**应当**与 `TASKS.md` 中该任务的"输出"小节一致。若需扩展，**必须**先向用户说明并获得同意。
- **R2.3** — 默认**严禁触碰**的文件 / 目录（除非任务明确要求）：
  - `ComfyUI-XYZNodes/__init__.py` —— 仅 T01 / T02 等明确要求时才能动；
  - `ComfyUI-XYZNodes/node.py`、`prompt_library_node.py`、`js/node.js` —— 现有 XYZ 节点代码，**与 Gallery 子模块完全无关**，**严禁**修改；
  - `PROJECT_SPEC.md`、`ARCHITECTURE.md`、`TASKS.md`、`AI_RULES.md` —— 文档类文件，**严禁**未经请求即修改；
  - `gallery_data/` 下所有运行时数据文件（`gallery.sqlite`、`gallery_config.json`、`gallery_audit.log`、`thumbs/**`）—— 这些是**运行时产物**，**严禁**作为代码改动的目标；
  - 任何位于 `ComfyUI-XYZNodes/` 之外的文件 —— **严禁**改动 ComfyUI 主程序、其他插件、Python 环境配置。
- **R2.4** — **严禁**调整 `.gitignore`、`requirements.txt`、依赖版本，除非任务明确要求且已与用户确认。
- **R2.5** — **严禁**重命名、移动、删除任何已有文件，除非任务明确要求。
- **R2.6** — **严禁**新建任务清单未声明的目录；目录结构以 `ARCHITECTURE.md §1` 为准。

## 3. 禁止重构（No Unsolicited Refactor）

> 对应需求 3：**不允许重构已有代码（除非明确要求）**。

- **R3.1** — **严禁**在没有明确指令的前提下进行任何形式的"重构"，包括但不限于：
  - 抽取公共函数 / 基类 / 工具模块；
  - 重命名变量、函数、类、模块、表、列、路由；
  - 调整代码组织顺序、导入顺序、文件拆分 / 合并；
  - 替换库（例如把 `requests` 换成 `httpx`）、替换数据结构、替换设计模式；
  - 把 `if/else` 改成多态、把多态改成 `if/else`；
  - 修改既有的日志格式、异常类型、错误码。
- **R3.2** — **严禁**"顺手"修复与当前任务无关的"代码味道"、Lint 警告、类型注解缺失、未使用导入。
- **R3.3** — **严禁**"现代化"现有代码（例如：旧式 `% formatting` 改 f-string、`os.path` 改 `pathlib`），除非任务明确要求。
- **R3.4** — 如果在阅读现有代码时发现**真实 bug**，**必须**只做记录并向用户报告，**严禁**未经请求即顺手修复，更**严禁**把"修 bug"和"做任务"混在同一个 diff 里。
- **R3.5** — 唯一允许的"轻微调整"是：为接入新代码而**必需**的最小钩子（例如在 `__init__.py` 中加一行 `gallery.setup(...)`），且**必须**在 PR 描述里单列一节解释。

## 4. 最小化改动（Minimal Diff）

> 对应需求 4：**所有改动必须最小化（minimal diff）**。

- **R4.1** — 每次提交的 diff **必须**满足"删一行就破功能"的标准：每一行新增 / 删除 / 修改都直接服务于当前任务的验收标准。
- **R4.2** — **严禁**包含以下"噪音"修改：
  - 仅调整空白、缩进、换行、引号风格、行尾分号；
  - 仅重排导入顺序；
  - 仅添加 / 删除 / 调整无信息含量的注释（例如 `# import xxx`、`# define function`）；
  - 自动格式化整个文件（只允许格式化**本次实际改动**的行）。
- **R4.3** — 新增代码遵循**最小可用实现（Minimum Viable Implementation）**：
  - 不写"将来用得上"的参数；
  - 不写未被任何路径调用的辅助函数；
  - 不写比验收标准更严格的校验（例如任务只要求接收 `int`，**严禁**额外加 `> 0` 校验）；
  - 不写比验收标准更宽松的校验（同样**严禁**）。
- **R4.4** — 单个任务的代码改动**应当**控制在合理体量内：
  - 新增 Python 文件：单个文件正常情况下 **≤ 300 行**；
  - 修改既有文件：单次任务对单个文件的净增 **≤ 200 行**；
  - 超出时**必须**先与用户确认是否拆分任务。
- **R4.5** — 注释只解释"**为什么**"或"非显而易见的约束"，**严禁**叙述"**做了什么**"。违反 R4.5 的注释视为噪音 diff。
- **R4.6** — 任何"为了让 diff 干净"而把无关行一起改掉的行为都**严禁**；保留原样反而是更小的 diff。

## 5. 严守 PROJECT_SPEC.md（Spec is Truth）

> 对应需求 5：**必须严格遵守 PROJECT_SPEC.md**。

- **R5.1** — `PROJECT_SPEC.md` 是**事实来源**。所有功能行为（FR-*）、非功能指标（NFR-*）、约束（C-*）**必须**逐字对齐。
- **R5.2** — **严禁**变更下列内容（除非用户**显式**要求修改 SPEC 本身）：
  - 路由路径（如 `/xyz/gallery/images`）、HTTP 方法、请求 / 响应字段名、字段语义；
  - 数据库表名、列名、约束、索引名；
  - 默认值（页大小 200、防抖 250ms、心跳 30s 等）；
  - UI 文案、按钮标签、控件 ID（FR 表中给出的 label）；
  - 文件 / 目录名（`gallery/`、`gallery_data/`、`gallery.sqlite`、`thumbs/` 等）；
  - 行为语义（例如"过滤组合是 AND"、"游标分页而非 OFFSET"、"软删除为否 = 物理删除"等）。
- **R5.3** — 当现有代码与 SPEC 不一致时：
  - **默认以 SPEC 为准**；
  - 但**严禁**自作主张把现有代码"修正"到 SPEC —— **必须**先问用户：是改代码、还是改 SPEC、还是当前任务暂不处理。
- **R5.4** — 当 SPEC 留有"未规定"空白时（例如错误码、日志级别、字段顺序），**必须**先问，**严禁**自由发挥。
- **R5.5** — `ARCHITECTURE.md` 描述的模块边界（`routes / service / repo / db / indexer / metadata / metadata_sync / thumbs / watcher / ws_hub / folders / paths / vocab`）**必须**严格遵守：
  - **严禁**让 `routes.py` 直接执行 SQL；
  - **严禁**让 `service.py` 直接读 / 写 SQLite 连接；
  - **严禁**让任何模块绕过 `repo.WriteQueue` 直接发起写操作；
  - **严禁**让 `metadata.py` 知道 SQLite 的存在；
  - **严禁**让 `thumbs.py` 调用 `routes.py`；
  - 模块依赖方向以 `ARCHITECTURE.md` 为准，**严禁**反向依赖、循环依赖。
- **R5.6** — **严禁**引入 `PROJECT_SPEC.md` / `ARCHITECTURE.md` 未提及的**第三方依赖**。需要新依赖时**必须**先问。

## 6. 禁止一次性实现完整系统（No Big Bang）

> 对应需求 6：**不允许一次性实现完整系统**。

- **R6.1** — **严禁**在单次回复中跨越多个 `TASKS.md` 任务 ID。即使用户表述含糊（例如"把 Gallery 做出来"），**必须**先把请求拆回到具体任务并逐一确认。
- **R6.2** — **严禁**输出"完整骨架代码包"（例如一次性给出 routes / service / repo / db / indexer 的全部空壳）。骨架**必须**严格落在某个任务（典型为 T01）的输出范围内。
- **R6.3** — **严禁**"一次性写完所有路由"、"一次性建完所有表"、"一次性导出所有前端组件"。表结构按 `TASKS.md` 的 Schema v1 → v2 → v3 顺序逐版本演进。
- **R6.4** — 数据库 Schema **必须**通过 `db.py` 的迁移框架按版本推进，**严禁**直接拼出最终态 DDL。每次任务只允许新增对应版本的迁移脚本。
- **R6.5** — 前端组件按任务粒度交付，**严禁**在 T11（SPA shell）阶段就把 `VirtualGrid` / `DetailView` / `BulkBar` 一并写出来。
- **R6.6** — 单次任务的产物**必须**可独立测试、可独立验收。如果某个改动"必须配合下一个任务才能跑通"，说明任务边界划错了 —— **必须**停下提问，而非强行实现。

## 7. 性能约束优先（Performance Is Non-Negotiable）

> 对应需求 7：**必须优先保证性能约束（5000+ 图片不卡顿）**。
> 规模目标：`PROJECT_SPEC §1` —— **2 000 – 50 000 张** ≥ 1024×1024 图片，平滑 UX。

- **R7.1** — 任何代码变更**必须**通过下列"性能体检"才能提交，违反任一条即视为不合格：
  - **后端列表 / 过滤接口**：**必须**走游标分页（cursor on `(sort_key, id)`），**严禁** `LIMIT ... OFFSET ...`（NFR / `PROJECT_SPEC §943-948`）；
  - **后端查询**：**必须**走已建索引或 FTS5；**严禁**对 5 万行做全表扫描、`LIKE '%xxx%'` 全列匹配、`SELECT *` 后在 Python 里过滤；
  - **后端写入**：**必须**通过 `repo.WriteQueue` 串行化，**严禁**在请求线程里直接 `conn.execute(INSERT/UPDATE/DELETE)`；
  - **后端 IO**：**严禁**在事件循环 / 请求处理线程里做同步磁盘 IO（读 PNG、生成缩略图、扫描目录）—— 必须丢到对应的 worker / 进程池；
  - **缩略图**：**严禁**在请求中**生成**缩略图阻塞响应；按需生成时**必须**走任务队列，并以 LIFO 视口优先（`ARCHITECTURE §4.9`）；
  - **前端列表**：**必须**虚拟滚动（仅渲染视口 + 缓冲行），**严禁**一次性把 N 个 `<img>` / `<div>` 全挂到 DOM；
  - **前端图片**：列表中**严禁**直接加载原图；**必须**使用 `/thumbnail` 端点的缩略图；详情页才允许加载原图；
  - **前端请求**：输入类过滤器（name / prompt / tag）**必须**防抖（≥ 250ms，对齐 `FR-3a`），**严禁**每个 keypress 直发请求；
  - **前端状态**：**严禁**在每次滚动 / 每次 keypress 上做 `O(N)` 的全列表遍历（N = 当前 working set 大小）；
  - **WebSocket**：**严禁**对每张图片单独广播；批量事件**必须**合并下发（参见 `Coalescer`、`ARCHITECTURE §4.10`）。
- **R7.2** — 任何引入 `O(N)` 以上复杂度（N = 图片总数）的逻辑**必须**在回复中显式列出复杂度分析，并由用户确认。
- **R7.3** — **严禁**为了"代码更整洁"而牺牲性能（例如用 ORM 全对象映射代替原生 SQL、把分页接口改成 `fetchall()` 后切片）。
- **R7.4** — 任何缓存（内存 LRU、磁盘缓存、autocomplete 缓存）**必须**有明确的容量上限与淘汰策略，**严禁**写"无界字典"。
- **R7.5** — 涉及性能敏感路径（indexer / thumbs / repo / VirtualGrid）的改动，**应当**附带粗略的"在 5 万张规模下最坏情况估算"，例如"这次改动会在冷启动时对每张图额外做 1 次 stat → 5 万次 stat ≈ X 秒，可接受/不可接受"。
- **R7.6** — 当任务要求与性能约束冲突时（例如某个 UI 行为天然 O(N²)），**必须**停下提问，**严禁**先实现再说。

## 8. 不确定即提问（Ask, Don't Guess）

> 对应需求 8：**如果不确定需求，必须先提问而不是猜测**。

- **R8.1** — 出现以下任何情况，AI **必须**停止执行、向用户提出明确问题、等待回复后再继续：
  1. 任务 ID 不明（用户没说做哪个任务，或同时提到多个）；
  2. 任务的"输入 / 输出 / 测试"在 `TASKS.md` 中不足以唯一确定实现；
  3. 实现方式与 `PROJECT_SPEC.md` / `ARCHITECTURE.md` 存在多种合理解读；
  4. 现有代码与文档不一致；
  5. 需要新建白名单之外的文件；
  6. 需要引入新依赖；
  7. 需要修改公共契约（路由 / 表 / 字段 / 事件名 / 配置 schema）；
  8. 估计改动规模超过 R4.4 的上限；
  9. 实现方案在性能上存在风险（违反 §7 任意一条）；
  10. 用户的口语化描述与文档术语不能 1:1 对应（例如"刷新"到底指 watcher、心跳、还是前端 reload）。
- **R8.2** — 提问**必须**满足：
  - 一次问完该批次所有疑点（**严禁**"挤牙膏式"问一句做一点）；
  - 给出**具体选项**（A/B/C），并标注每个选项的影响（性能、复杂度、与 SPEC 的对齐度）；
  - 标明 AI 的**默认倾向**及理由，但**严禁**在用户回答前按倾向动手。
- **R8.3** — **严禁**用"我猜你的意思是 X，所以我直接做了 X"开场。猜测必须以**问题**形式提出，而不是以**已完成的代码**形式提出。
- **R8.4** — 当用户回答"你看着办" / "随便" / "你决定"时，AI **应当**：
  - 选择**最保守、最贴合 SPEC、最小风险**的方案；
  - 在回复开头明确写出"按 X 方案执行，理由：…"；
  - 不要把"自由裁量"理解为"可以越界做更多事"。

## 9. 交付格式（Delivery Format）

每次完成（或部分完成）任务时，AI 的回复**必须**包含下列结构化小节，缺一不可：

1. **任务 ID**：明确标注本次实现对应 `TASKS.md` 中的哪个 ID。
2. **文件白名单**：本次实际**新建 / 修改**的文件列表（完整路径）。
3. **未改动声明**：明确声明"本次未触碰下列文件 / 模块"，至少覆盖 §2 中的默认禁区。
4. **验收对账**：逐条引用 `TASKS.md` 中该任务的"测试"条目，标注 ✅ 已满足 / ⚠️ 部分满足（说明原因）/ ❌ 未做（说明原因）。
5. **未实现 / 故意省略清单**：明确列出"按 R1.2 / R4.3 没做的、留给后续任务的"内容。
6. **遗留问题 / 待确认**：列出在实现过程中发现但**未自行处理**的问题（参考 R3.4）。
7. **性能影响估算**（仅在 §7 涉及的模块改动时）：粗略给出 5 万规模下的最坏情况。
